# flakectl User Guide

## Overview

flakectl is an AI-powered CI failure classifier. It analyzes GitHub Actions
workflow runs that have failed, reads the logs, and categorizes each failure
into one of several root-cause types:

- **Flaky tests** -- intermittent failures caused by timing, race conditions,
  or fragile assertions
- **Infrastructure flakes** -- CI environment issues such as network errors,
  registry outages, or image pull failures
- **Real bugs** -- genuine code defects correctly caught by tests
- **Build errors** -- compilation or build-system failures

The business purpose is to reduce the time developers and SREs spend manually
triaging CI failures. Instead of re-running pipelines to see if a failure was
"just a flake," teams can consult the report and focus on real issues.

### Key features

- Automated log download and analysis for every failed job in a workflow run
- Root-cause classification with descriptive category names
- Fix correlation: links each root cause to recent commits and open PRs that
  may address it
- Structured output in markdown (`report.md`), JSON (`report.json`), and
  plain text (`summary.txt`)
- Runs as a GitHub Action or standalone CLI

## How it works

flakectl uses Claude AI agents (via the
[Claude Agent SDK](https://docs.anthropic.com/en/docs/agents-sdk)) to perform
the analysis. The pipeline has six stages:

1. **Fetch** -- queries the GitHub API for failed workflow runs
2. **Progress** -- builds a coordination file listing every failed run and its jobs
3. **Classify** -- launches one Claude agent per run; each downloads logs,
   searches for failure patterns, and writes a classification
4. **Correlate** -- a separate agent searches recent commits and open PRs for
   fixes matching each root cause
5. **Extract** -- parses all classifications into `report.md` and `report.json`
6. **Summarize** -- generates a 2-3 sentence `summary.txt` from the report

All category names, descriptions, root-cause summaries, fix correlations, and
the summary text are generated by AI.

## Limitations

### Contextual understanding

The AI analyzes log text and, when available, test source code. It may
misinterpret domain-specific patterns, custom test frameworks, non-standard CI
setups, or logs in unexpected formats. Classifications are based on textual
patterns, not deep understanding of your application's semantics.

### Read-only operation

flakectl only reads and analyzes -- it does not fix tests, modify code, create
issues, or take any remedial action. It is an informational tool.

### Possibility of inaccuracies

The AI may produce:

- **Incorrect classifications** -- labeling a real bug as a flake, or vice versa
- **Inaccurate root-cause descriptions** -- misidentifying the underlying cause
  of a failure
- **Hallucinated details** -- generating plausible-sounding but incorrect
  explanations
- **Wrong fix correlations** -- linking a commit or PR that is unrelated to the
  actual failure, or missing the actual fix

### Fix correlation is heuristic

The correlation between root causes and commits/PRs is based on textual
matching and AI judgment. Listed fixes may not actually address the failure,
and actual fixes may be missed. The report distinguishes between "match"
(high confidence) and "possible" (lower confidence) correlations.

## AI-generated content in the report

The following parts of the report are AI-generated:

| Report element | AI-generated? | Example |
|----------------|---------------|---------|
| Category name | Yes | `test-flake/hooks-lifecycle-timeout` |
| Description | Yes | "Flake: a lifecycle test intermittently times out..." |
| Example summary | Yes | Detailed analysis paragraph for each root cause |
| Fix correlation | Yes | Links to commits/PRs that may fix the issue |
| Summary text (`summary.txt`) | Yes | "18 failed runs analyzed: 18 caused by flakes..." |
| Run metadata (IDs, dates, branches) | No | Fetched directly from GitHub API |
| Job names and step names | No | Fetched directly from GitHub API |
| Error messages | No | Extracted verbatim from CI logs |

### Example

Below is an annotated excerpt from a typical report. Parts marked with
**(AI)** are generated by the AI:

```
### 1. `test-flake/hooks-lifecycle-timeout`              <-- (AI) category name

**Description:** Flake: a lifecycle test intermittently   <-- (AI) description
times out waiting for a device state transition.

- **Failed runs:** 9                                      <-- from GitHub API
- **Failed jobs:** 12                                     <-- from GitHub API
- **Test IDs:** 12345, 12346                              <-- (AI) extracted from logs
- **Fix(-es):**
  - 2026-02-18 [#42](...) Fix lifecycle timeout           <-- (AI) fix correlation
- **Example error:** `Timed out after 5m waiting for...`  <-- verbatim from logs
- **Example summary:** The test waits for a device        <-- (AI) analysis
  state change callback that races with ...
```

## Reviewing AI-generated content

**Always review AI-generated content prior to use.**

Before acting on flakectl's classifications:

- Verify that the category assignment matches the actual failure by checking
  the linked CI run logs
- Confirm that listed fix correlations are relevant to the root cause
- Cross-check descriptions against your knowledge of the codebase and
  test infrastructure

Follow your organization's guidelines on use of AI-generated content.

## Monitoring and quality signals

Every report includes built-in quality signals in the `execution_stats`
section of `report.json`:

### Agent execution statistics

| Metric | Description |
|--------|-------------|
| `classifier_agents.count` | Number of classifier agents launched |
| `classifier_agents.errors` | Agents that crashed or timed out |
| `classify_phase.turns` | LLM round-trips per classifier (min/max/avg/median) |
| `recheck_phase.turns` | Recheck round-trips per classifier |
| `*.duration_ms` | Wall-clock time per phase (min/max/avg/median) |
| `*.tool_calls` | Tool usage breakdown (Read, Grep, Edit, download_log, etc.) |
| `correlator_agent` | Fix-correlation agent stats |
| `summarizer_agent` | Summary generation agent stats |
| `*.usage` | Token usage from the Claude API (see below) |

The `usage` field is passed through from the Claude API. The core token
fields (`input_tokens`, `output_tokens`, `cache_read_input_tokens`,
`cache_creation_input_tokens`) are part of the documented API. Additional
fields may be present depending on the API version. Example:

```json
"usage": {
  "input_tokens": 343,
  "cache_creation_input_tokens": 14661,
  "cache_read_input_tokens": 196234,
  "output_tokens": 3576,
  "service_tier": "standard"
}
```

### Classification quality signals

| Signal | Location | What it indicates |
|--------|----------|-------------------|
| `unclear_runs` | `report.json` | Runs the AI could not confidently classify |
| `unfinished_runs` | `report.json` | Runs that failed to complete (agent crash, timeout) |
| `model` | `metadata` | Claude model used -- detect drift from model updates |
| `flakectl_version` | `metadata` | Tool version -- correlate quality changes with releases |
| Error rate | `errors / count` | Agent reliability |

### Periodic validation process

To validate that flakectl's classifications remain accurate:

1. **Review feedback submissions** from the
   [feedback form](https://forms.gle/BteSPY4zoZoeLcwg9) for reported
   inaccuracies
2. **Spot-check recent reports** -- pick 2-3 categories and verify against
   actual CI logs
3. **Monitor trends** -- compare unclear_runs, unfinished_runs, error rates,
   and turn counts across reports. A sustained increase may indicate a problem
4. **Check after model or version changes** -- review the next report more
   carefully for regressions

Recommended frequency: after each model or tool update, and quarterly
otherwise.
